The reporter asked for this bug to be reconsidered as to whether it could be a security bug. I explained in

 but I can expand slightly why this does not meet our requirements.



<span class="line"><!----> but I can expand slightly why this does not meet our requirements.<!----></span>



<br>



<br>





When a page embeds an iframe, it has very wide latitude to control how the iframe is drawn on the page though a range of attributes, layout options, transforms, and filters. This does have security implications because the embedder can manipulate user interactions with a cross-origin embedded frame for purposes such as ad fraud -- such uses are called UI redress attacks, aka 'clickjacking'.



<span class="line"><!---->When a page embeds an iframe, it has very wide latitude to control how the iframe is drawn on the page though a range of attributes, layout options, transforms, and filters. This does have security implications because the embedder can manipulate user interactions with a cross-origin embedded frame for purposes such as ad fraud -- such uses are called UI redress attacks, aka 'clickjacking'.<!----></span>



<br>



<br>





The potential for such attacks is inherent to the composable design of the web. We can't prevent that kind of manipulation, so a bug like this that adds another way to do it makes no difference to user safety.



<span class="line"><!---->The potential for such attacks is inherent to the composable design of the web. We can't prevent that kind of manipulation, so a bug like this that adds another way to do it makes no difference to user safety.<!----></span>



<br>



<br>





Instead we provide mechanisms for web developers to protect themselves from UI redress attacks. There are HTTP headers (X-Frame-Options and CSP frame-ancestors [1]) that sites can use to prevent their content from being embedded within cross-origin iframes. For content that is designed specifically to be embeddable, Chrome implements v2 of the IntersectionObserver API [2]. If you find a bug that can circumvent the scope of protection provided by any of these clickjacking mitigations, we would consider that a security flaw.

For content that is designed specifically to be embeddable, Chrome implements v2 of the IntersectionObserver API [2]. If you find a bug that can circumvent the scope of protection provided by any of these clickjacking mitigations, we would consider that a security flaw.<!----></span>
