<span class="line"><!---->Assume a local home network with a public DNS for the public facing IP, and NAT-ed internal network (quite the usual setup).<!----></span>

<span class="line"><!---->Some ISP routerboxes (in this case Orange FR/Livebox) catch all traffic targeting the public IP from the internal network (192.168.* &gt; public : goes straight to the router box). It fails also to apply NAT/PAT-rules on the traffic in this scenario.<!----></span>



<br>

<span class="line"><!---->The "alternative" is to use a hosts file to redirect the local traffic to the internal IP instead of the public IP. Or to use a local DNS with views to redirect the DNS to the internal IP.<!----></span>

<span class="line"><!---->However, DoH configuration on Chrome messes this up. With no real alternative:<!----></span>

<span class="line"><!---->With DoH, chrome is trying (legitimately) reaching the public facing IP, because this is the actual record.<!----></span>

<span class="line"><!---->But by doing that, and failing to read the hosts entry values, it gets catched by the routerbox.<!----></span>

<span class="line"><!---->Having the ability to keep using hosts entries (or another alternative ?) would avoid this.<!----></span>

<span class="line"><!---->Turning off DoH is also not a "good" solution as for a mobile client it means having it disabled inside and OUTSIDE of the local home network.<!----></span>
